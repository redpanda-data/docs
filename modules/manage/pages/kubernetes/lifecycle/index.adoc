After your Redpanda cluster is running in Kubernetes, you can manage broker lifecycle operations such as grouping brokers into node pools, safely removing brokers, recovering from failures, and performing rolling restarts for maintenance.

== Node pools

The NodePool custom resource lets you manage groups of Redpanda brokers as independent units within a single cluster. Each NodePool creates its own StatefulSet, giving you control over broker placement, resources, and lifecycle for blue/green migrations and vertical scaling.

See xref:manage:kubernetes/k-node-pools.adoc[Node Pools].

== PVCUnbinder

The PVCUnbinder is an emergency backstop for when a node running a Redpanda Pod suddenly goes offline. It detects the lost node, retains the associated PersistentVolume, and removes the PersistentVolumeClaim so the Pod can be rescheduled on a healthy node without losing data.

See xref:manage:kubernetes/k-nodewatcher.adoc[Enable the PVCUnbinder].

== Decommission brokers

Decommissioning is the safe and controlled way to remove a broker from the cluster. The process migrates partition replicas to remaining brokers before shutting down the broker, so you avoid data loss or cluster instability.

See xref:manage:kubernetes/k-decommission-brokers.adoc[Decommission Brokers].

== Recovery mode

Recovery mode starts Redpanda with limited functionality so you can repair a failed cluster. In recovery mode, you can inspect and manage partitions while most client-facing operations are disabled.

See xref:manage:kubernetes/tiered-storage/k-recovery-mode.adoc[Recovery Mode].

== Rolling restart

A rolling restart restarts one broker at a time while the remaining brokers continue running. This minimizes downtime during configuration updates, version upgrades, or cluster maintenance.

See xref:manage:kubernetes/k-rolling-restart.adoc[Rolling Restart].
